What is this?

Over six weeks, I built a personalised program management toolkit using Claude Code (Anthropic's AI coding assistant). It connects to Jira, Confluence, Slack and Glean, and produces EG-branded outputs: decks, dashboards, status reports, meeting briefs, and more. Everything here is repeatable. You can build your own.

The key insight: The magic is not in the AI itself. It is in the "memory files" you build over time: your design system, your report templates, your program context. These turn generic AI outputs into production-quality work that matches your team's standards.

Program context

Separate memory files store program details: phases, financials, stakeholders, milestones, risks

➔
Step 3

One prompt

"Create a SteerCo deck for 8 May" and Claude combines template + context + latest Jira/Confluence data

➔
Step 4

Branded .pptx

Output is an EG-branded PowerPoint file ready to share. Edit in PowerPoint as normal for final tweaks.

Critical learning: The template memory file is what makes or breaks deck quality. Without it, every AI-generated deck looks like generic clip art. With it, stakeholders cannot tell the difference between hand-built and generated slides. It took a few iterations to get right, but once encoded, every future deck benefits.
Practical note: Claude uses python-pptx (a Python library) to build decks. It works best for structured content: tables, KPI cards, timelines, matrices. For highly visual or illustrative slides, you will still want to polish in PowerPoint. The 80/20 rule applies: Claude builds the structure, you add the finishing touches.

📚

Memory Files

Markdown files that persist across sessions. They store your design system, program context, stakeholder details, template rules, and lessons learned. Claude reads these at the start of every conversation, so it already knows your world.

⚡

Custom Skills

Reusable command templates (slash commands). Each one defines: what to do, what sources to check, what format to output. They encode your workflow so you do not have to explain it every time.

Model Context Protocol servers that connect Claude to your tools: Jira, Confluence, Slack, Glean, Trino. Claude queries them directly instead of you copy-pasting between tabs.

Lessons Learned

Six weeks of building this toolkit was not smooth sailing. Here is what I wish I had known from day one, and what to expect if you start building your own.

The first week is rough. That is normal.

First 3 decks looked terrible

Without a template memory file, Claude used wrong fonts, wrong colours, wrong positioning. Every deck looked like generic clip art. It took 3-4 iterations of correcting and encoding rules before decks started looking professional.
Fix: Invest time upfront encoding your design system. Once done, every future deck benefits. The ROI compounds fast.

Context disappears between sessions

I kept re-explaining my program, my stakeholders, my preferences. Every session felt like starting over. It was frustrating and slow.
Fix: Write your MEMORY.md and program context files. This was the single biggest unlock. Claude reads them at session start and already knows your world.

Outputs need editing, not just accepting

Early on I treated outputs as final. They were not. Claude gets 80% right and needs you to refine the last 20%. That is the workflow: generate fast, polish to quality.
Fix: Treat Claude like a very fast first-drafter. Your expertise is still needed for accuracy, tone, and judgement. The time saving is in the drafting, not the thinking.

Corrections need to be saved

I would correct something (wrong colour, wrong terminology, wrong date format) and then Claude would make the same mistake next session. Corrections in conversation are ephemeral.
Fix: When you correct something, ask Claude to save it to memory. "Remember: always use RMD not GP." This makes corrections permanent.
Technical lessons

Excel: use COM, not openpyxl

The Python library openpyxl destroys merged cells, conditional formatting, and complex formulas. Lost a workback schedule to this. Now all Excel work uses Win32 COM automation (talks to Excel directly).
Rule: If your Excel file has merged cells, formulas, or conditional formatting, always use COM. It is slower but safe.

PowerPoint: Layout 114 (Blank) is your friend

I tried using template layouts with pre-positioned placeholders. They fight you. The title appears mid-page, text boxes overlap. Much better to use Blank layout and build everything from scratch with exact coordinates.
Rule: Blank layout + custom title bar rectangle + positioned content = full control.

Jira API cannot update rich-text fields

Spent time trying to programmatically update "Status Update" and "Next Steps" fields on Program issue types. The API rejects these. They must be updated in the Jira UI.
Rule: Use the API for structured fields (dates, assignee, status transitions). Use the UI for rich-text narrative fields.

HTML dashboards beat SharePoint for speed

I considered building dashboards in SharePoint or Power BI. Too much overhead for a one-person program. Single-file HTML with localStorage gives you interactivity, persistence, and zero dependencies. Open in any browser, works offline.
Rule: Start with HTML. Move to a proper tool only if you need multi-user collaboration or live data refresh.
Process and mindset lessons

Build incrementally, not all at once

I did not plan all 15 commands upfront. I built each one when I hit a task for the third time. "I keep doing this manually" = time to automate it. The toolkit grew organically from real needs.

Memory files are living documents

They are not "set and forget". I update program context files after every major meeting or decision. Stale memory = stale outputs. 2 minutes of updating saves 20 minutes of re-explaining next time.

Be specific in your prompts

"Make me a deck" produces mediocre results. "Create a 6-slide SteerCo deck for 8 May with KPI overview, phased roadmap, RAID table, and actions slide" produces excellent results. Specificity is the lever.

The compound effect is real

Week 1: slow, frustrating, lots of corrections. Week 3: noticeably faster, fewer corrections. Week 6: 32 outputs in a single week. The investment in memory files and skills pays off exponentially over time.

Do not try to replace your judgement

Claude handles the admin, drafting, formatting, and data-pulling. You still make the decisions, set the strategy, and manage the relationships. The toolkit amplifies your capacity, it does not replace your expertise.

Share early, get feedback

I showed colleagues a generated deck after week 2. Their reactions ("wait, that's not hand-built?") validated the approach and gave me ideas for what to build next. Do not wait until it is perfect.
The biggest mistake I see people make: They try Claude once, get a generic output, and conclude "it does not work for my job." The difference between a generic output and a production-quality one is context. If you give Claude the same context you would give a new team member (who you are, what you are working on, what good looks like), the outputs transform. The first 30 minutes of setup is the barrier. Everything after that compounds.
Getting Started: Build Your Own Toolkit

You do not need to be technical to do this. I am not an engineer. Here is the path I would recommend, based on what worked for me:

Prerequisites

1. Claude Code installed (ask IT or check the internal tooling channel)
2. Node.js installed (Claude Code needs this to run)
3. ~30 minutes for initial setup
4. A willingness to iterate: your first outputs will not be perfect, but they improve fast
The 5-step path
  • 1
    Create your MEMORY.md
    Start with the basics: your name, role, team, active programs, and key preferences (formatting rules, terminology, tools you use). This is the file Claude reads at the start of every session. It is the foundation for everything else.
  • 2
    Add one program context file
    Pick your most active program. Create a markdown file with: program overview, current phase, key stakeholders, milestones, risks, and recent decisions. Claude will use this context to generate relevant, accurate outputs instead of generic ones.
  • 3
    Try your first dashboard or deck
    Ask Claude to generate a Weekly Recap dashboard or a simple overview deck. The first attempt will need corrections. That is normal. Each correction teaches Claude your preferences, and you can save those as memory for next time.
  • 4
    Connect your tools (MCP)
    Set up MCP connectors for Jira and Confluence (these have the most impact for PMs). Once connected, Claude can pull live data instead of you copying and pasting. Glean and Slack are valuable additions when you are ready.
  • 5
    Build your first custom command
    Pick your most repetitive task (morning check-in, meeting prep, status report) and create a skill file for it. A skill is just a markdown file that tells Claude: what to do, what sources to check, and what format to output. Once saved, you trigger it with a slash command.
Common questions

Do I need to know how to code?

No. I am not an engineer. Claude writes the code (Python for decks, HTML/CSS/JS for dashboards). You describe what you want in plain English. That said, over time you will start to understand what is happening under the hood, and that helps you ask for more precise things.

How long does setup take?

The basic setup (MEMORY.md + one program file) takes about 30 minutes. After that, it is incremental: you add memory files as needs arise. Within a week of normal use, you will have a solid foundation. Within a month, it becomes genuinely hard to work without it.

Is it safe to use with program data?

Follow the same data handling policies you use for any tool. Memory files are stored locally on your machine. MCP connectors use your existing authentication (SSO). Claude does not retain data between sessions beyond what is in your local memory files.

What if the output is wrong?

It will be sometimes, especially early on. Treat Claude like a very fast junior PM: it needs context and corrections. The key is that corrections become permanent via memory files. You teach it once, and it remembers for every future session.

Can I share my memory files with colleagues?

Absolutely. Template files (like the EG PowerPoint design system) are fully shareable. Program context files would need to be adapted. The skill files (commands) are also shareable and reusable across team members.

What is the biggest ROI?

For me: PowerPoint generation (hours saved per week), meeting prep (better prepared, less anxiety), and the morning pulse (never miss an overdue action). Your mileage will vary depending on where you spend the most time on admin.
